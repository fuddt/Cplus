# 教材タイトル案
**C++で学ぶゲームロジック設計入門**  
〜バイオ風ハーブ回復システムを作る〜

---

## 全体構成（カリキュラム設計）

### 第0章：ゴールの確認
完成形イメージの共有：
- プレイヤーがHPを持つ
- ハーブを使うと回復する
- 最大HPを超えない
- HP割合で状態（Condition）が変わる
- ロジックは適切に分離されている

### 第1章：まずは動く最小構成を作る
#### 1-1 Playerクラスの作成
**学ぶこと：**
- `struct` と `class` の違い
- メンバ変数とメンバ関数
- カプセル化と状態更新

**実装課題：**
- メンバ変数: `hp`, `maxHp`
- メンバ関数:
    - `heal()`: 回復処理
    - `damage()`: ダメージ処理
    - `updateCondition()`: HPに応じた状態更新
- ※ この段階ではアイテムクラスは作成しない。

### 第2章：設計を考える（超重要）
設計の問いかけを通じて、責務分離の概念を導入：
- HPを直接変更していいのか？（カプセル化）
- どこに回復ロジックを書くべきか？
- 状態更新はどこで実行するのが最適か？
- **図解：** クラス図やシーケンス図を用いて、責務の分離を理解させる。

### 第3章：Herbクラスを作る
#### 3-1 単純版（継承なし）
まずは具体的なクラスから作成する：
```cpp
struct Herb {
    int healAmount = 30;

    void use(Player& player) {
        player.heal(healAmount);
    }
};
```
**学ぶこと：**
- 参照渡し (`&`)
- クラス間の依存関係

### 第4章：インベントリを導入
`std::vector` を用いてアイテム管理を導入：
- `std::vector<Herb>`
- `useItem(index)`

**学ぶこと：**
- STL (Standard Template Library)
- コンテナの基礎

### 第5章：設計の拡張（課題の提示）
さらなる機能拡張を想定し、設計の限界を提示：
- ハーブが複数種類あったら？（グリーン、レッド、ミックス…）
- 鍵（Key）などの別カテゴリのアイテムを追加するなら？
- **問い：** `std::vector<Herb>` の中に「鍵」は入れられるか？

### 第6章：王道設計へ進化（ポリモーフィズム）
インターフェース（抽象クラス）の導入：
```cpp
class Item {
public:
    virtual void use(Player& player) = 0;
    virtual ~Item() = default;
};
```
**学ぶこと：**
- 純粋仮想関数
- 多態性（ポリモーフィズム）
- **参照 (`Item&`)** による共通操作

### 第7章：アイテムボックスと所有権（ポインタ）
「預ける・引き出す」操作を通じてポインタを学ぶ：
- インベントリとボックスの間でアイテムを移動させる。
- **学ぶこと：**
    - なぜポインタが必要か？（異なるサイズのオブジェクトを一つの配列で管理する）
    - スマートポインタ (`std::unique_ptr`)
    - 所有権の移譲 (`std::move`)
    - メモリの動的確保 (`std::make_unique`)

---

## 教材の哲学
1. **スモールステップ：** いきなり継承やポインタを使わない。
2. **課題解決型：** まず不便さを感じてから、それを解決するための設計パターンを学ぶ。
3. **ゲーム体験とのリンク：** 「預ける」「使う」といったゲーム的な操作と、C++の高度な概念（所有権、ポインタ）を紐付ける。
